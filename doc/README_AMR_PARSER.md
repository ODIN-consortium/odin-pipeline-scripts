# AMR Data Parser

## Overview

The `parse_amr_json.py` script processes Antimicrobial Resistance (AMR) JSON files generated by the Nextflow wf-metagenomics pipeline. It reads AMR data for specific samples defined in a metadata Excel file and converts the nested JSON structure into a tabular format (CSV/Excel/Feather) suitable for analysis.

## Input Arguments

- `--data_path` (required): Root data path containing the `nanopore_processed` directory with AMR output files
- `--metadata_excel` (required): Path to the metadata Excel file that defines which samples to process
- `-o, --output` (required): Output directory where the resulting files will be saved (`amr_data.csv`, `amr_data.feather`, `amr_data.xlsx`)
- `--show-summary` (optional): Display summary statistics about the parsed data

## Logic for Finding Connected Run Accessions

The script uses a comprehensive approach to handle samples that may be sequenced across multiple runs:

1. **Check metadata for connections**: For each run accession, the script identifies connected runs by matching three criteria:
   - Same `sample_id` (e.g., "SITE01_WW_001")
   - Same `sampling_date` (samples collected on the same day)
   - Same `barcode` (same barcode used during sequencing)
   
   All three must match for runs to be considered connected.

2. **Directory priority based on connections**:
   - **If connected runs exist**: The script searches for merged data first, prioritizing:
     1. `sampleName` directory (most complete merged data)
     2. `merged_{run_accession}` directory (run-specific merged data)
     3. Individual `run_accession` directory (fallback)
   - **If no connected runs**: It searches for individual data first:
     1. Individual `run_accession` directory
     2. `sampleName` directory
     3. `merged_{run_accession}` directory

3. **Marking incomplete data**: If a sample has connected runs but data is only found in an individual run directory (not merged), it's flagged as `incomplete_data=True`. This indicates the sequencing runs haven't been merged yet.

This ensures that if multiple sequencing runs exist for the same sample (same sample_id, date, and barcode), the script prefers to use combined/merged data when available.

## JSON to DataFrame Conversion

The AMR JSON files have a nested structure that gets flattened into a table:

### JSON Structure (Input)
```json
{
  "barcode01": {
    "pass": true,
    "results": {
      "gene_description_1": {
        "count": 5,
        "meta": [
          {
            "RESISTANCE": "antibiotic1;antibiotic2",
            "SEQUENCE": "ATCG...",
            "START": 100,
            "END": 500,
            "%COVERAGE": 95.5,
            "%IDENTITY": 99.2
          }
        ]
      }
    }
  }
}
```

### DataFrame Structure (Output)

Each resistance gene detection becomes one row:

| run_accession | barcode | pass_status | gene_description | resistance | short_name | count | coverage_percent | identity_percent | sample_id | ... |
|---------------|---------|-------------|------------------|------------|------------|-------|------------------|------------------|-----------|-----|
| ERR123456 | barcode01 | True | gene_description_1 | antibiotic1;antibiotic2 | anti;anti | 5 | 95.5 | 99.2 | SITE01_WW_001 | ... |

### Key Transformations

1. **Nested loops flatten the structure**: The parser iterates through barcodes → genes → metadata entries
2. **Resistance splitting**: Multi-drug resistance entries (semicolon-separated) are kept together in one row, with a `multi_resistance` flag set to `True` when multiple resistances are present
3. **Short names**: A `short_name` field is created by taking the first 4 characters of each resistance type
4. **Metadata enrichment**: Each row is enriched with sample information (sample_id, sample_code, sampling_date, protocol_id, sequencing_kit_id, alias, type, runName, sampleName, etc.) from the metadata Excel
5. **Geographic data**: Site coordinates (latitude/longitude) and country are added by matching the `sample_id` to a sites reference table
6. **Incomplete data flagging**: Rows are marked with `incomplete_data=True` if they come from an individual run when merged data should exist

## Output Files

The script generates three equivalent files in different formats:
- `amr_data.csv`: Standard comma-separated values
- `amr_data.feather`: Fast binary format for pandas (used for incremental updates)
- `amr_data.xlsx`: Excel spreadsheet
- `amr_data_summary_by_resistance.csv`: Aggregated summary showing counts and statistics per resistance type

### Incremental Updates

The script supports incremental updates by merging with existing data:
- If `amr_data.feather` already exists in the output directory, the script loads it
- Existing data for the run_accessions being processed is removed
- New data is appended and deduplicated
- This allows you to update specific samples without reprocessing everything

## Example Usage

```bash
python scripts/parse_amr_json.py \
  --data_path /path/to/data \
  --metadata_excel metadata/nanopore_samples.xlsx \
  --output results/amr_analysis \
  --show-summary
```

## Summary Statistics

When `--show-summary` is provided, the script displays:
- Total number of AMR records
- Number of unique run_accessions and barcodes processed
- Number of unique resistance types found
- Pass status distribution (True/False)
- Top 5 most common resistance types
- Resistance records per barcode
- Average coverage and identity percentages

## Data Flow

1. **Read metadata**: Load nanopore metadata Excel file and filter to rows with barcodes
2. **For each row**: 
   - Check for connected runs (same sample_id + sampling_date + barcode)
   - Determine directory search order based on connections
   - Find AMR JSON files in the appropriate directory
3. **Parse JSON files**: Extract resistance data and flatten nested structure
4. **Enrich data**: Add metadata columns and geographic information
5. **Merge with existing**: Combine with previous results if available
6. **Export**: Save to CSV, Feather, and Excel formats
7. **Summarize**: Generate aggregated statistics by resistance type
